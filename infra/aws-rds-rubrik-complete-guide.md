# AWS RDSバックアップ完全ガイド！Rubrikで通信費と脅威を防ぐ設計術

## 導入
AWSのデータベース（Amazon RDS / Aurora）を運用する中で、こんな悩みを抱えていませんか？

「AWSネイティブのバックアップだけでは、特権アカウントが乗っ取られた時に全データが消去されるリスクがある…」
「かといって、オンプレミスにデータを引き抜くと、莫大な通信費（エグレスコスト）がかかってしまう…」
「開発用に論理ダンプ（SQLファイル）が必要だが、手作業での抽出が面倒だ…」

AWSの標準機能は便利ですが、高度なランサムウェア攻撃や内部不正からデータを完全に守り切るには限界があります。

この記事では、ゼロトラストデータ管理のグローバルリーダーである**Rubrik（ルーブリック）**を活用し、**AWS RDSのバックアップを「エグレスコストの罠」を回避しながら完全自動化・保護するアーキテクチャ**を解説します。

AIOpsやSREとしてインフラ運用を高度化したいエンジニア向けに、APIを使った自動化スクリプトや、導入時にハマりやすい注意点まで具体的に解説します。

---

## 1. AWSネイティブバックアップ（スナップショット）の限界

AWSには標準でRDSのスナップショット機能があります。それにも関わらず、エンタープライズ企業がサードパーティ製品であるRubrikを導入する最大の理由は**「特権ID侵害に対する復元力の確保」**です。

AWSアカウントの管理者権限（Rootや強力なIAM）が奪取された場合、同一アカウント内にあるRDSのスナップショットは、攻撃者によって簡単に削除（DROP）されてしまいます。

Rubrikは、AWSのネイティブAPIをコントロールプレーンから呼び出し、単なるスナップショットを**「変更不可・削除不可（イミュータブル）な真のバックアップ」**へと昇華させるアーキテクチャを採用しています。

---

## 2. 致命的な落とし穴！エグレスコストによる「クラウド破綻」を防ぐ戦略

ランサムウェア対策として「クラウドのデータを毎日オンプレミスの物理機器にバックアップする」という設計をよく耳にします。しかし、AWS RDSにおいてこれは**絶対にやってはいけないアンチパターン**です。

数TB〜数PBあるデータベースを毎日インターネット経由でオンプレに転送すると、AWSのデータ転送費（Egressコスト：通常 $0.09/GB）が爆発し、通信費だけでクラウドの運用コストが完全に破綻してしまいます。

Rubrikはこの「通信費の罠」を痛いほど理解しており、**「クラウドのデータはオンプレに引き抜かない」**というアーキテクチャを基本としています。では、どうやってコストを抑えつつデータを安全に隔離するのでしょうか？

### ① 「オンプレ」ではなく「同一リージョン内の隔離領域」へ逃がす
RubrikはRDSのデータをオンプレミスに持ってくるのではなく、**「Rubrik Cloud Vault」**と呼ばれるSaaS型の隔離ストレージへ転送します。

ここで重要なのが、**「本番RDSと同じAWSリージョン内」**のVaultを保存先に指定することです。AWSの仕様上、同じリージョン内であれば、別アカウント（Rubrik側の管理アカウント）へのデータ転送であっても、高額なEgressコストを劇的に最小化（またはゼロに）することができます。

### ② 永久増分（Incremental-Forever）による転送量の極小化
DR（ディザスタリカバリ）目的で、どうしてもAWSからAzure上のVaultなどへデータを引き抜きたい場合もあるでしょう。
その場合でも、Rubrikは最初の一回（フルバックアップ）だけ通信費がかかりますが、2回目以降は**「変更されたブロック（増分データ）」のみ**を転送・合成します。これにより、日々のランニングコストをしっかり抑え込みます。

### ③ AIOps的アプローチ「コストとリスクの階層化（Tiering）」
すべてのRDSを無理に隔離領域へ引き抜く必要はありません。Rubrikの「SLAドメイン（ポリシー）」を使って、以下のようにメリハリをつけるのが最も賢い運用です。

*   **Tier 1（超重要DB：顧客の決済情報など）**
    *   **戦略:** コストをかけてでも、AWS外（Rubrik Cloud Vault）へデータを完全に引き抜いて隔離・保護する。
*   **Tier 2（通常DB：社内のログシステムなど）**
    *   **戦略:** 通信費はかけず、AWSアカウント内でのスナップショット管理にとどめる。ただしRubrikからAPI経由で「Retention Lock（削除禁止）」だけをかけ、論理的な保護を行う。

> **【ポイント】**
> 現代のデータ保護の正解は**「データは元のクラウドに置いたまま、管理権限とアカウント境界だけを切り離す」**ことです。これで通信費をハックしつつセキュリティを担保できます。

## 3. なぜRDSは「ウイルススキャン」ができないのか？

VM（仮想マシン）やファイルサーバーのバックアップでは、ファイルの暗号化を検知したり、マルウェアをスキャンしたりするのが一般的です。しかし、RDSに対してはこれらのアプローチは機能しません。

理由は、RDSのスナップショットが「AWS独自のブロックレベルイメージ」であり、ホストOSも隠蔽されているため、ファイル単位でのスキャンが構造的に不可能だからです。

### 代替策：DSPM（データセキュリティ態勢管理）によるリスク可視化
ウイルススキャンができない代わりに、RubrikはクラウドDB特有の脅威にフォーカスした**DSPM機能**を提供しています。

これは、Rubrikがバックグラウンドで一時的なリソースを立ち上げ、DB内のテーブルを直接スキャンする機能です。
**「個人情報（PII）やクレジットカード情報が、どのDBのどのテーブルに保存されているか」**を自動で分類・マッピングし、データ漏洩時のリスク範囲を正確に把握できるようにします。

## 4. 【実践コード付】Rubrik APIで論理バックアップを自動化

RubrikはRTO（目標復旧時間）を最優先にしているため、ブロックレベルのスナップショットから直接 `pg_dump` や `mysqldump` のような論理ダンプをダウンロードするネイティブ機能はありません。

しかし、AIOpsエンジニアであれば、**豊富なREST APIを活用して論理ダンプ抽出を完全自動化**できます。

以下のPythonコードは、「一時的なRDSインスタンスを作成する」ためのAPIリクエスト例です。CI/CDパイプラインに組み込むことで、手作業をゼロにできます。

```python
import requests
import json
import os

RUBRIK_NODE_IP = "rubrik.example.local"
API_TOKEN = os.getenv("RUBRIK_API_TOKEN")
HEADERS = {
    "Authorization": f"Bearer {API_TOKEN}",
    "Content-Type": "application/json"
}

def create_temp_rds_for_dump(snapshot_id, target_subnet_group, target_sg):
    """論理ダンプ抽出用の一時RDSインスタンスをエクスポート（作成）する"""
    
    url = f"https://{RUBRIK_NODE_IP}/api/v1/aws/rds_instance/snapshot/{snapshot_id}/export"
    
    # 抽出作業用のため、マルチAZを無効化し安価なインスタンスクラスを指定
    payload = {
        "targetInstanceId": "temp-dump-db-instance",
        "instanceClass": "db.t3.medium",
        "isMultiAz": False,
        "subnetGroupId": target_subnet_group,
        "vpcSecurityGroupIds": [target_sg]
    }
    
    response = requests.post(url, headers=HEADERS, data=json.dumps(payload), verify=False)
    response.raise_for_status()
    
    task_info = response.json()
    print(f"Export Job Started. Task ID: {task_info['id']}")
    return task_info['id']

# 実行例（IDは環境に合わせて動的に取得）
SNAPSHOT_ID = "11111111-2222-3333-4444-555555555555"
SUBNET_GROUP = "isolated-dump-subnet-group"
SECURITY_GROUP = "sg-0123456789abcdef0"

task_id = create_temp_rds_for_dump(SNAPSHOT_ID, SUBNET_GROUP, SECURITY_GROUP)
# ※この後、Taskの完了を検知して pg_dump を実行し、完了後にDBをAPIで削除する
```

## 5. 導入前に知っておくべき運用上の注意点（Gotchas）

実際にRubrikでRDSを保護する際、エンジニアがハマりやすい技術的な落とし穴をまとめました。

*   **KMSキーの権限不足：** 暗号化されたRDSを別のアカウントやリージョンにリストアする場合、リストア先環境のIAM Roleに、元データのKMSキー（CMK）の `kms:Decrypt` 権限などがないとエラーになります。
*   **AWS APIのレート制限（Throttling）：** 数百のRDSスナップショットを同時に処理しようとすると、AWS側のAPI制限に引っかかることがあります。バックアップの開始時間を分散させる設定が必要です。
*   **ダンプ抽出時の「消し忘れ」によるコスト増：** APIで一時DBを作成してダンプを抽出した後、DBを削除（Teardown）する処理がパイプラインで失敗すると、AWS側で無駄なコンピュート料金が発生し続けます。エラーハンドリングは確実に行いましょう。

## 6. よくある質問（FAQ）

**Q. AWS Backupとの違いは何ですか？**
A. AWS Backupは強力ですが、特権IDが乗っ取られるとバックアッププランごと破壊されるリスクがあります。Rubrikは「別アカウント・別システムからの隔離と不変性（イミュータビリティ）」を担保する点で、セキュリティレベルが異なります。

**Q. オンプレミスのデータベースも一緒に管理できますか？**
A. はい。SaaSであるRubrik Security Cloudから、AWS上のRDSだけでなく、オンプレミスのVMwareや物理DB（Oracle, SQL Server）も単一の画面・単一のポリシーで統合管理できます。

## 7. まとめ：RDSの保護は「隔離」と「自動化」が鍵

この記事の要点を整理します。

1.  **エグレスコスト対策:** オンプレに引き抜かず、同一リージョンのCloud Vaultへ隔離することで通信費を回避する。
2.  **コストの階層化（Tiering）:** 重要度に応じて「引き抜き」と「論理ロック」を使い分ける。
3.  **リスクの可視化:** ウイルス検知ではなく、DSPMで機密データの所在を特定する。
4.  **運用の自動化:** APIを使って一時DBの生成と論理ダンプ抽出を自動化し、工数を削減する。

Rubrikを導入することで、単なる「バックアップ運用」から、インシデント発生時の復旧を確約する「サイバーレジリエンスの確保」へと、インフラ運用のレベルを一段引き上げることが可能です。
