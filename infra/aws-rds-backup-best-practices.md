# 【記事①】完全版：RubrikでAWS RDSをバックアップする構成ベストプラクティス

## 導入
「オンプレミスで使っているRubrikを使って、AWS上のRDSも一元管理したい」
「でも、クラウドのバックアップ設計はオンプレと違うと聞いて不安…」

インフラ責任者の皆様、その直感は**大正解**です。

AWSのバックアップをオンプレミスと同じ感覚で設計すると、**「本番DBのフリーズ」や「毎月数百万円の通信費請求（クラウド破産）」**という大惨事を招きます。

この記事では、大規模環境（数十TB〜）を想定した、Rubrik×AWS RDSの絶対に失敗しないベストプラクティス（最適構成）を完全解説します。

この記事を読めば、上司や経営層に自信を持って安全・安価なクラウドインフラ設計を提案できるようになります。

## 1. 構成のベストプラクティス 3つの鉄則

結論から言います。以下の3つの鉄則を守るだけで、クラウドバックアップの失敗の9割は防げます。

1.  **データはAWSから出さない（地産地消）**
2.  **別アカウントを「金庫（Vault）」にする**
3.  **本番DBはマルチAZかAuroraにする**

それぞれ、なぜ必要なのかを解説します。

## 2. 鉄則①：データはAWSから絶対に出さない

オンプレミスの感覚だと「バックアップデータは手元（自社サーバー）にダウンロードして保管したい」と考えがちですが、AWSでこれをやると破滅します。

### 恐ろしい「下り通信費（Egress）」の罠
AWSは「データを入れるのは無料、外に出すのは超高額」というルールがあります。
もし200TBのバックアップデータをオンプレミスに引き出した場合、**1回で約270万円（約18,000ドル）のデータ転送料**が請求されます。

**【解決策】**
バックアップデータは、AWSの中（同一リージョン内）に留めてください。同一リージョン内なら通信費は「無料」です。

## 3. 鉄則②：別アカウントを金庫（Vault）にする

「AWSの中に置いておいたら、アカウントがハッキングされた時にバックアップごと消されるのでは？」

その対策として、Rubrikの**クロスアカウントバックアップ**を使用します。

本番環境（アカウントA）とはIAM（管理者権限）が完全に切り離された**「バックアップ専用の別アカウント（アカウントB）」**を作成し、そこへRubrikから自動でデータを転送します。
この隔離された別アカウントが、ランサムウェアからデータを守る「最強の金庫（Vault）」となります。

## 4. 鉄則③：本番DBは「マルチAZ」か「Aurora」に

RubrikがRDSのバックアップ（スナップショット）を取得する瞬間、数秒〜数分間、DBの読み書きが一時停止します。

これを防ぐためには以下のどちらかの構成が必須です。

* **RDS（MySQLやPostgreSQLなど）：** 常に裏で予備機を動かす「マルチAZ構成」にする。バックアップは予備機から取られるため、本番は止まりません。（※維持費は2倍になります）
* **Amazon Aurora：** クラウドに最適化されたDB。記憶領域（ストレージ）が分離しているため、1台構成であってもバックアップ時にシステムが止まりません。

## よくある質問（FAQ）

**Q. Rubrik Cloud Vaultという製品は使わなくていいの？**
A. はい。RDSの保護においては、わざわざ外部のRubrik Cloud Vaultへ送る必要はありません。AWS内の「別アカウント」をVaultとして使うのが最も安価で安全なベストプラクティスです。

**Q. AWS標準機能だけでバックアップするのと何が違うの？**
A. Rubrikを導入することで、「オンプレとAWSの1画面での統合管理」「ランサムウェア感染時のAIによる異常検知」「復旧リハーサルの全自動化」など、エンタープライズに必須のセキュリティ機能が追加されます。

## まとめ：正しい設計図でクラウド運用を

RubrikとAWSを組み合わせたバックアップのベストプラクティスは以下の通りです。

* **管理**はRubrik（RSC）で一元化する。
* **データ**はAWS内の「クロスアカウント」に隔離する。
* **通信費**をゼロにするため、同一リージョン内で完結させる。
