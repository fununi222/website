# AWS KMSの料金を99%削減する｜エンベロープ暗号化による最強のコストハック

</figure>

「AWSのデータをKMSで暗号化したら、サーバー代を上回るAPI利用料が請求された……」

本格的なクラウド運用を開始した企業が、セキュリティを高めた結果として直面するのが、この**「KMSの課金トラップ」**です。セキュリティレベルを一切落とさず、この高額請求を月額わずか数ドルに抑え込む手法が存在します。

本記事では、コスト効率と鉄壁のセキュリティを両立させる魔法の設計、**「エンベロープ暗号化」**の真髄を解説します。

---

## 1. なぜKMSで「クラウド破産」が起きるのか？

AWS [KMS](https://fununi222.github.io/website/html/glossary/system-glossary.html#:~:text="KMS")（Key Management Service）の料金体系には、恐ろしい罠が潜んでいます。

*   **鍵の保管料**: 月額約1.5ドル（安価）
*   **API呼び出し料**: 1万回あたり $0.03（**これが致命傷になる**）

「1万回で数セントなら安い」と錯覚してはいけません。もしシステムが「DBに1行書き込むたびにKMSを直接呼び出す」設計（直接暗号化）になっていた場合、月間10億リクエストのシステムでは**毎月3,000ドル（約45万円）以上**が青天井で請求されます。

---

## 2. 解決策：エンベロープ暗号化による「API通信の極小化」

自前の鍵管理サーバーを立てる必要はありません。AWS標準の仕組みだけでコストを劇的に下げるベストプラクティスが、**「エンベロープ（封筒）暗号化」**です。

### 仕組みのメタファー：封筒とマスターキー
1.  **直接暗号化**: 毎回、大きな重い金庫（KMS）へデータを運び、鍵をかけてもらう（通信量大・課金大）。
2.  **エンベロープ暗号化**: 金庫（KMS）から「使い捨ての小さな鍵（データキー）」だけをもらい、自分の手元でデータを暗号化する。そして、その「小さな鍵」を金庫の鍵で暗号化した「封筒」に入れてデータと一緒に保管する。

### なぜ99%安くなるのか？
システムは発行された「データキー」をメモリ内に数分〜数時間キャッシュして再利用します。
**KMSと通信するのは「鍵を発行する最初の一回」だけ**になるため、背後で100万回の暗号化が行われても、KMSへの課金は「1回分」で済みます。

---

## 3. 実装の要：AWS Encryption SDKの活用

エンベロープ暗号化を自前で実装（暗号化アルゴリズムの選定やパディング処理）するのは、脆弱性を生むリスクが高いため推奨されません。

AWSが提供する公式ライブラリ **[AWS Encryption SDK](https://docs.aws.amazon.com/ja_jp/encryption-sdk/latest/developer-guide/introduction.html)** を使用しましょう。これを使えば、以下の高度な処理が数行のコードで実装可能です。

*   データキーの自動生成と暗号化
*   セキュアなキャッシュ管理（Data Key Caching）
*   複数のKMSキー（マルチリージョン）による冗長化

---

## 4. 補足：混乱しがちな「3つのVault（金庫）」を整理する

暗号化の設計において、データを隔離する「Vault」という言葉は3つの文脈で使われます。これらを混同すると設計ミスに繋がります。

| Vaultの種類 | 用途 | 特徴 |
| :--- | :--- | :--- |
| **HashiCorp Vault** | 資格情報・秘密情報の管理 | 汎用的な秘密管理。大量データの暗号化には不向き。 |
| **[Rubrik Cloud Vault](https://fununi222.github.io/website/html/glossary/system-glossary.html#:~:text="Rubrik%20Cloud%20Vault")** | バックアップデータの隔離保管 | 大規模データの保存に特化したクラウド貸し倉庫。 |
| **AWS Backup Vault** | AWS内での論理的隔離 | RDS等のバックアップを別アカウントへ逃がす仕組み。 |

---

## 5. まとめ：賢いアーキテクチャこそが「利益」を生む

1.  **直接暗号化はアンチパターン**: リクエスト数に比例する課金は、事業の成長をコストが阻害する原因になる。
2.  **エンベロープ暗号化の導入**: AWS Encryption SDKを活用し、KMSのAPI呼び出し回数を劇的に削減する。
3.  **キャッシュとセキュリティのバランス**: データキーの有効期限を適切に設定し、コストとリスクの最適解を見つける。

セキュリティを「コスト」として捉えるのではなく、**「設計の工夫で最小化できる変数」**として捉え直すことが、クラウド時代のエンジニアに求められる知性です。
