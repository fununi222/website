# Rubrikのスケーリング戦略：クラスタ追加とノード追加のベストプラクティス

データ量が増加し、[Rubrik](https://fununi222.github.io/website/html/glossary/system-glossary.html#:~:text="Rubrik")インフラの拡張を検討する際、多くのインフラ管理者が直面する疑問があります。

**「リソース制限が細かくできないなら、負荷を分散させるために新しいクラスタを都度追加していくべきか？」**
それとも、**「既存のクラスタにノードを追加して巨大化させるべきか？」**

結論から言うと、**物理的・地理的な制約がない限り、圧倒的に「ノード追加（単一クラスタの巨大化）」が正解**です。
この記事では、Rubrikの裏側で動いている最新のデータベースアーキテクチャを紐解きながら、クラスタ分割がなぜダメなのか、正しいスケーリングのベストプラクティスを解説します。

## なぜ「クラスタの都度追加」はアンチパターンなのか？

従来の古いバックアップソフトの感覚だと、システムが大きくなりすぎると不安定になるため、小さなシステムを複数並べる（サイロ化する）のが安全だと考えられていました。しかし、Rubrikでこれを行うと以下の致命的なデメリットが発生します。

* **重複排除効率の劇的な悪化：** クラスタを分けると、データ削減の辞書（重複排除ドメイン）が分断されます。単一クラスタなら1回分の容量で済んだデータが、クラスタごとに保存されてしまい、ストレージ容量を激しく浪費します。
* **リソースプールの分断：** 余剰リソース（CPUやメモリ）がクラスタ間でシェアできないため、片方は過負荷、もう片方はスカスカという非効率な状態に陥ります。
* **運用コストの増大：** 管理する画面とポリシーが分散し、トラブルシューティングの手間が倍増します。

## メタデータストアの進化：CockroachDBが無限の拡張を可能に

「でも、1つのクラスタを大きくしすぎると、検索やメタデータの処理が遅くなるのでは？」という心配があるかもしれません。

実は、初期のRubrikはメタデータ管理に「Apache Cassandra」というデータベースを使っており、大規模になるとデータが偏ったり、検索が遅延する問題がありました。
しかし現在のRubrikは、金融機関などでも使われる先進的な分散データベース**「[CockroachDB](https://fununi222.github.io/website/html/glossary/system-glossary.html#:~:text="CockroachDB")」**へアーキテクチャを刷新しています。これにより、ノードをどれだけ追加しても、パフォーマンスが劣化することなくリニアに性能が向上する「無限のスケールアウト」が可能になりました。

システムが巨大化して不安定になるトラウマは、もはや過去のものです。Rubrikは、ノードを追加すればするほど、障害への耐性も並列処理のスピードも向上する設計になっています。

## クラスタ追加が「正解」となる3つの例外シナリオ

基本はノード追加ですが、例外的に「新しいクラスタを作るべき」ケースも存在します。

### 1. 物理的・地理的に離れている場合
海外の支社や、遅延の大きい別データセンターのデータを保護する場合、1つのクラスタで同期させるのは不可能です。拠点ごとにクラスタを構築するのが鉄則です。

### 2. 完全なエアギャップ（隔離）環境を作りたい場合
ランサムウェア対策として、本番環境から完全に切り離されたDR（ディザスタリカバリ）環境を作る場合です。
2025年の最新の調査（Ransomware Recovery Council調べ）によると、不変（イミュータブル）なバックアップと、ゼロトラストの分離環境を持つ企業は、ランサムウェア攻撃からのリカバリ成功率が99.7%に達しています。このような高度な要件には、独立したターゲットクラスタが必須です。

### 3. 法令による厳密なハードウェア隔離
テナントごとに「物理的なハードウェアレベル」でのデータ分離が法律や契約で義務付けられている場合です（小規模ならRubrik Edgeなどの仮想アプライアンスが適しています）。

## 複数クラスタになってしまった場合の救世主：Rubrik Security Cloud (RSC)

どうしても複数のクラスタを運用しなければならない場合、管理の煩雑さを解決するのが**「[RSC](https://fununi222.github.io/website/html/glossary/system-glossary.html#:~:text="RSC") (Rubrik Security Cloud)」**です。

RSCは、世界中にあるRubrikクラスタを、SaaS型の1つの画面で統合管理できるプラットフォームです。今後はオンプレミスの個別画面ではなく、RSCによる統合管理がRubrikの標準となります。

## まとめ

* 負荷分散のための「クラスタ追加」は、ストレージ効率と管理性を下げる悪手。
* 基本は**「単一クラスタへのノード追加（スケールアウト）」**が絶対のベストプラクティス。
* 地理的隔離や高度なランサムウェア対策が必要な場合のみ、クラスタを分ける。
