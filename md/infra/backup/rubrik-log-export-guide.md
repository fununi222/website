# 【完全ガイド】Rubrik脅威監視で検体を特定！詳細ログの取得方法と仕組み

## 導入
Rubrikの脅威監視ダッシュボード（シアターモニター）で、悪意のある検体を発見できた！
これは素晴らしい成果です。

しかし、次にこんな壁にぶつかっていませんか？

「詳細なログはどうやって取得すればいいの？」
「画面のデータをエクセルにまとめるために、GUIをHTML化してスクレイピングするしかないの？」

結論から言えば、GUI画面のスクレイピングは絶対に不要です。

この記事では、Rubrik Security Cloud（RSC）におけるログ取得の仕組みと、なぜ画面のHTML化が不要なのかを初心者にも分かりやすく解説します。

## 1. Rubrikの脅威監視（Threat Monitoring）とは？
まずは前提知識の整理です。

Rubrikの脅威監視は、単なるバックアップソフトの機能ではありません。
取得したバックアップデータの中に、ランサムウェアなどのマルウェア（検体）が潜んでいないかを自動でスキャンする機能です。

システムが検体を発見すると、GUIのダッシュボード上に警告が表示されます。
あなたが画面上で「検体を取得（特定）できた」と認識したのは、この高度な分析プロセスが無事に完了した証拠です。

## 2. 結論：GUIのHTMLスクレイピングは不要！
多くの方が、「画面に表示されているデータを抽出するには、ブラウザのHTMLを読み取ってスクレイピングするしかない」と誤解しています。

しかし、Rubrikは**「APIファースト」**と呼ばれる設計で作られています。

APIファーストとは、「画面（GUI）で出来る操作は、すべて裏側のプログラム（API）でも全く同じように出来る」という設計思想です。

つまり、画面に見えている検体データやログは、システムから直接、綺麗なデータ（CSVやJSON）としてダウンロードできるのです。

## 3. なぜHTML化（スクレイピング）はNGなのか？
HTMLスクレイピングには致命的な弱点があります。

*   **すぐ壊れる：** メーカー側が画面のデザイン（CSSやボタンの位置）を少し変更しただけで、プログラムがエラーを起こします。
*   **緊急時に使えない：** いざサイバー攻撃を受けて急いでログが欲しい時に、スクリプトが動かないという大事故に繋がります。

セキュリティ運用において、このような不安定な手法を採用することは非常に危険です。

## 4. 詳細ログはどこにあるのか？
検体が特定されると、Rubrikは自動的にシステムログを生成します。

例えば、「分析が完了した」「ハッシュ値が一致した」といったイベントは、個別のログとして記録されます。
これらのログは、Syslog転送を使って既存のセキュリティ監視システム（SIEMなど）に直接送ることが可能です。

### メリット・デメリット
| 手法 | メリット | デメリット |
| :--- | :--- | :--- |
| **HTMLスクレイピング** | 直感的に目に見えるものをそのまま取ろうとする | 画面の仕様変更ですぐに壊れる。動作が遅い。運用保守の手間が膨大。 |
| **API/Syslog活用** | 仕様変更に強い。100%正確なデータが即座に取れる。他システムと簡単に連携できる | 最初にAPIの仕組みを少し学ぶ必要がある |

## 5. 具体例（リアルケース）
実際に、高度なセキュリティ分析ツールである「Google Chronicle」などを利用している企業では、RubrikからのログをJSON形式で直接取り込んでいます。

画面をコピペするスタッフを雇うのではなく、システム同士を会話させることで、深夜に検体が発見されても自動でログが抽出される環境を作っているのです。

## FAQ（よくある質問）
**Q. 検体が発見された時の具体的なイベント（ログ）の名前は何ですか？**
A. 例えば、分析が成功した場合は`ThreatMonitoringAnalysisSucceeded`、検体（一致）が発見された場合は`ThreatMonitoringHashCatalogAnalysisMatchesFound`といった明確なイベント名が記録されます。

Rubrikで検体を発見した後、詳細なログの取得は「確実に可能」です。
GUI画面をHTML化してスクレイピングするような不安定な手法は避けましょう。

RubrikはAPIファースト設計であり、画面に出る情報はすべてシステムから直接抽出できます。
