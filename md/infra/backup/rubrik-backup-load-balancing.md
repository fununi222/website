# Rubrikのバックアップ負荷分散ガイド！リソース制限ができない時の対策

[Rubrik](https://fununi222.github.io/website/html/glossary/system-glossary.html#:~:text="Rubrik")を導入してバックアップを自動化したものの、**「バックアップ中に本番ストレージの負荷が100%に張り付いてしまう」「Rubrik側でネットワークの帯域制限（スロットリング）が設定できずに困っている」**と悩んでいませんか？

実は、Rubrikにはバックアップ取得時のトラフィックを直接制限するネイティブ機能がありません。これは仕様上の欠陥ではなく、Rubrikが「極限まで早くデータを保護する」という思想で作られているためです。

この記事では、Rubrik環境における本番ストレージへの過負荷問題（Noisy Neighbor問題）を解決し、安全に負荷を分散させるための具体的な対策とベストプラクティスを解説します。

## なぜRubrikはバックアップのリソース制限（スロットリング）ができないのか？

結論から言うと、ネットワークの方向（IngressとEgress）によって、Rubrikの制限機能が異なるためです。

* **バックアップ時（Ingress：入ってくるデータ）：直接制限不可**
* **アーカイブ・レプリケーション時（Egress：出ていくデータ）：精密な制限が可能**

Rubrikが仮想マシン（VM）やデータベースからデータを吸い上げる際、Rubrikはあくまで「受信側」です。ネットワークの仕組み上、受信側から送信元（本番ストレージ）の送信速度を強制的に制限することは困難です。

Rubrikは「可能な限り最短時間でバックアップを終わらせて、データを安全な場所に確保する」というアプローチをとります。そのため、高性能なフラッシュストレージ環境などでは、Rubrikが全力でデータを読みに行き、結果として本番ストレージのI/Oリソースを食い潰してしまう事象が発生します。

## ストレージの過負荷を防ぐ4つの具体策

Rubrik自体で制限ができない以上、**インフラ側の機能や間接的なチューニング**を組み合わせるのがプロのベストプラクティスです。

### 1. OSレベルでのQoS（帯域制限）の設定

Windows ServerやRHEL（Red Hat Enterprise Linux）などの物理サーバーを保護する場合、OSの機能を使ってネットワーク帯域を制限します。
たとえば、Rubrikエージェントのアウトバウンド通信を「最大1Gbpsまで」とOS側でキャップをかけることで、ストレージへの過剰なアクセスを物理的に防ぎます。

### 2. ネットワークスイッチ側のトラフィックシェーピング

VMware vSphere環境の場合、バックアップデータは専用のvmkernelポートを通ります。
このポートが接続されている物理スイッチや、分散仮想スイッチ（vDS）の機能を使って、ポート単位での帯域幅制限（トラフィックシェーピング）を行うのが効果的です。

### 3. Rubrikの「詳細設定」で並列数を減らす

Rubrikの管理画面から帯域自体は絞れませんが、同時に走る処理の数は減らせます。
「保護の管理」ウィザードの最下部にある「詳細設定」から、「チャネル数」や「マルチセクションバックアップ」の値を小さく調整してください。同時ストリーム数が減るため、ストレージへの負荷が間接的に下がります。

### 4. ストレージアレイのスナップショット連携

最もスマートな解決策は、ハイパーバイザー（VMwareなど）経由での読み込みをやめ、本番ストレージ自体が持つスナップショット連携機能を活用することです。これにより、バックアップの負荷をストレージコントローラにオフロードできます。

## 【注意】タスク同時実行数の隠れた制限について

実は、Rubrikの各ノードには、システムを安定稼働させるための「同時実行数のハードリミット」が隠されています。
デフォルトでは、**「1つのESXiホストにつき最大3タスク」「1つのRubrikノードにつき最大3タスク」**しか同時にバックアップを実行できません。

一見するとリソースが余っているように見えても、この制限に引っかかってバックアップが待ち状態になり、SLA（目標時間）に間に合わなくなるケースがあります。大規模環境でキューが詰まる場合は、Rubrikサポートに連絡してバックエンドからこの制限値を調整（例：3→4などへ引き上げ）してもらう必要があります。

## まとめ：負荷分散のために「クラスタを追加」するのは正解か？

バックアップの負荷が制御しづらいため、「それなら小さなクラスタを複数作って、物理的に負荷を分けよう（クラスタの都度追加）」と考えるエンジニアは少なくありません。

しかし、**これはRubrikの強みを完全に殺してしまう「アンチパターン（やってはいけない運用）」です。**
