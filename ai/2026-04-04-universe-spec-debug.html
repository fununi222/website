OpenClaw Network Troubleshoot

VPS 環境におけるパケットフィルターの罠と解決策

// XServer VPS 上で OpenClaw の外部通信が失敗した事象を、インフラ起点で切り分けた実戦ログ

1. 発生した現象と環境

  • 対象: OpenClaw v2026.4.1（XServer VPS / Ubuntu）
  • 現象: openclaw skills check ではツールが Ready に見えるが、browser-js の外部アクセスがタイムアウト / fetch failed。
  • 補足: サーバー内の curl は無反応、nslookup だけ成功。

2. 主因：VPS 固有のパケットフィルター仕様

今回のボトルネックは OS やモデル能力ではなく、VPS 事業者側のインフラフィルターでした。XServer VPS のパケットフィルターが ON の場合、許可ポート以外は双方向で遮断されます。

  • DNS（UDP 53）が通る理由: 管理用途で 53 番が既定許可されるケースが多いため。
  • HTTP/HTTPS（TCP 80/443）が失敗する理由: 外向きリクエストの戻りパケットも明示許可が必要で、フィルターにより遮断されるため。

ローカル PC や家庭用ルーターのステートフル通信とは挙動が異なるため、VPS 特有の制約として扱う必要があります。

3. 解決ステップ

ステップ1：Scope 権限を解放

openclaw devices list
openclaw devices approve <REQUEST_ID>

ステップ2：DNS の安定化

echo "nameserver 8.8.8.8" > /etc/resolv.conf

ステップ3：インフラ・フィルターを開放（最重要）

  1. 管理パネルの「パケットフィルター」を一時的に OFF。
  2. または「TCP 80 / 443」を接続許可ポートへ追加。

4. 診断コマンド（切り分けの定番）

# DNS（UDP 53）
nslookup google.com

# TCP ハンドシェイク（80/443）
curl -v --connect-timeout 5 https://www.google.com

この2本をセットで見ることで、名前解決と外部通信のどちらが詰まっているかを高速に切り分けできます。

5. 結論

今回の失敗要因は LLM ではなく、純粋なネットワーク・インフラ制約でした。gemini-3.1-flash-lite-preview のような軽量モデルでも、インフラが整えば実運用レベルで十分にツール連携できます。

「SSH が開いている = 外部通信できる」ではない点は、VPS 構築で必ず押さえるべきチェックポイントです。
