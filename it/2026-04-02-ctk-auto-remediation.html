自動化に向けたCTK監視是正

VMware vSphere環境におけるバックアップトラブルを防ぐ。CBT（Changed Block Tracking）不整合の検知と是正を自動化する実践手法。

背景：なぜCBT/CTKは不整合を起こすのか

VMwareの増分バックアップに不可欠な機能であるCBT（Changed Block Tracking）は、仮想マシンの構成パラメータ（ctkEnabled）と実際の仮想ディスク（scsiX:Y.ctkEnabled）の不整合により、予期せず無効化されることがあります。

主な原因として、スナップショットの頻繁な作成・削除、VMの強制終了、あるいは一部の移行作業に伴う設定情報の不一致が挙げられます。これが起きると、バックアップソフトはデータの整合性を保つためにフルスキャン（事実上のフルバックアップ）にフォールバックし、バックアップ時間とストレージ容量を圧迫します。

技術スタック：PowerCLI × Jenkins

手動での確認を排除するため、PowerCLIを使用して全VMのCBTステータスを走査し、不整合（パラメータがFalse、あるいは欠落）を検知した場合に自動的にリセットするフローを構築しました。

Jenkinsを用いることで、毎日12:00（バックアップ実行前）の定期実行と、是正結果の通知を一元管理しています。

自動是正スクリプトの実装例

PowerCLI Script (CBT Reset logic) 
# CBTの不整合をチェックし、必要に応じてリセットする
$vmlist = Get-VM | Where-Object { $_.PowerState -eq "PoweredOn" }

foreach ($vm in $vmlist) {
    $cbtEnabled = $vm.ExtensionData.Config.ChangeTrackingEnabled
    if ($cbtEnabled -ne $true) {
        Write-Host "Resetting CBT for $($vm.Name)..."
        
        # CBTの再有効化プロセス
        $spec = New-Object VMware.Vim.VirtualMachineConfigSpec
        $spec.ChangeTrackingEnabled = $true
        $vm.ExtensionData.ReconfigVM($spec)
        
        # 設定を反映させるために一時的なスナップショットを作成・削除
        $snap = New-Snapshot -VM $vm -Name "CBT-Reset-Temp"
        Remove-Snapshot $snap -Confirm:$false
        
        Write-Host "CBT for $($vm.Name) has been restored."
    }
}

運用上の注意点と今後の展望

この自動是正は非常に強力ですが、高負荷なワークロード（DBサーバー等）に対しスナップショットを頻繁に作成・削除すると、スタン（一瞬のフリーズ）が発生する可能性があるため、実行タイミングには注意が必要です。今後はAIを活用し、バックアップの過去ログから不整合の兆候を予測検知する仕組みを検討しています。
